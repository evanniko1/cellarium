02

What actually spawned the containers

The shipped code path, unchanged. One docker run --rm per simulation.

This is the same path a normal cellarium.generate --parallel 8 run takes — no test-only shim in the dispatch. A thread pool of eight fans the (design × seed) matrix out; each worker blocks on its own container for the full simulation, so N parallel sims = N containers, each writing a distinct output directory.

05

Reading this honestly

What the test proves, and the two things it doesn't.

scope

Bounded by RAM, not cores. Each simulation loads ~1 GB of sim_data; eight fit comfortably in 128 GB here, but on a laptop the practical ceiling is 2–3 concurrent, not 8. The speedup is near-linear only while memory and cores are unsaturated.

metric

No doubling time at one generation. The manifest surfaces instantaneous growth_rate but not doubling_time for a single generation — a full division interval needs --generations 2+. Not a parallelism limitation.

Cellarium · reproducible via cellarium.generate --seeds 8 --generations 1 --parallel 8 against a local wcEcoli image. Model is Stanford academic-licensed and run locally; no model code or output is redistributed.
